# Collaboration Guide — Project 5: Mapping Emotions

This guide explains how to work together in GitHub Codespaces without overwriting each other's work. Follow these rules carefully — they exist to prevent **merge conflicts**, which happen when two people edit the same part of the same file at the same time.

---

## The Golden Rule

> **Each person owns specific files and sections. Edit only what is assigned to you.**

## Git Workflow in Codespaces

Follow this sequence every time you work on the project:

### Before You Start
```bash
git pull
```
Always pull first to get the latest changes before you start editing.

### While You Work
1. Edit **only your assigned files and sections**
2. Save your file (`Ctrl+S`)

### When You Finish a Task
```bash
git add <filename>
git commit -m "Member 2: filled in JMU chapters 3 and 4"
git push
```

Commit **frequently and in small chunks** — after finishing each chapter or each section. Small commits are much easier to merge.

### When Ready to Publish the Website
Run this from anywhere in the repo terminal:

```bash
bash project_5_mapping_emotions/publish.sh
```

This exports the notebook, assembles all website files into `docs/`, and pushes to GitHub Pages in one step. **Always `git pull` first** so you have everyone's latest changes before publishing.

## Order for Editing `team_data.json`

Because `team_data.json` is one file that everyone must touch, follow this order to avoid conflicts:

**4-member teams:**
1. Member 1 edits slot 1 → commits + pushes
2. Member 2 **pulls**, edits slot 2 → commits + pushes
3. Member 3 **pulls**, edits slot 3 → commits + pushes
4. Member 4 **pulls**, edits slot 4 → commits + pushes; then **deletes the slot 5 object** → commits + pushes

**5-member teams:**
1. Member 1 edits slot 1 → commits + pushes
2. Member 2 **pulls**, edits slot 2 → commits + pushes
3. Member 3 **pulls**, edits slot 3 → commits + pushes
4. Member 4 **pulls**, edits slot 4 → commits + pushes
5. Member 5 **pulls**, edits slot 5 → commits + pushes

Do not skip the pull step — it ensures you have the previous member's changes before you make your own.

## If You Get a Merge Conflict

A merge conflict means two people changed the same line. VS Code shows it like this in the file:

```
<<<<<<< HEAD
your version of the text
=======
their version of the text
>>>>>>> origin/main
```

### Steps to resolve it:
1. **Don't panic.** Open the file in VS Code.
2. Look for the `<<<<<<<`, `=======`, and `>>>>>>>` markers.
3. Decide which version to keep (usually both, combined carefully).
4. Delete the three marker lines entirely — the file should not contain `<<<<<<<`.
5. Save the file.
6. Run:
   ```bash
   git add <filename>
   git commit -m "Resolved merge conflict in [filename]"
   git push
   ```

### Best way to avoid conflicts in the first place:
- Stick strictly to your assigned sections
- Pull before you start every work session
- Commit and push frequently — don't let changes pile up overnight

## Common Mistakes to Avoid

| Mistake | Why It Causes Problems |
|---------|----------------------|
| Editing another member's section | Creates a conflict the next time they push |
| Forgetting to pull before editing | Your local copy is stale; now two versions exist |
| Making many changes before committing | One big commit is hard to merge cleanly |
| Editing `flythrough_template.html` | This file is shared and should never be modified |
| Re-ordering chapters in the config | Chapter order is shared; moving them causes conflicts |
